# 第5周 Day2：SQL窗口函数进阶

## 1. 今日学习内容

本次学习了：

- LAG获取上一期数据
- LEAD获取下一期数据
- 环比增长额
- 环比增长率
- NULLIF避免除以0
- 两个月移动平均
- 按渠道划分趋势窗口
- 上一可用月份与自然上月的区别
- CASE WHEN趋势状态分类
- 日期覆盖率对趋势分析的影响

## 2. 月度销售额数据范围

正式销售分析使用：

- 订单状态为已完成
- 销售额有效标记为1

月份分析还要求：

- 订单日期不为空

全部有效订单有15条，但日期有效订单只有12条，日期覆盖率为80%。

因此，月份趋势只能代表日期有效的12条订单。

## 3. LAG

LAG用于取得当前行之前的记录。

按月份排序后：

- 2月的上一行是1月
- 3月的上一行是2月
- 1月没有上一行

月度销售额：

- 2026年1月：2565.80
- 2026年2月：2912.05
- 2026年3月：6462.00

增长额：

- 2月比1月增加346.25
- 3月比2月增加3549.95

## 4. 环比增长率

环比增长率计算方式：

本月销售额减去上月销售额
→ 除以上月销售额
→ 乘以100%

结果：

- 2月环比增长率：13.49%
- 3月环比增长率：121.91%

1月没有上月数据，所以增长额和增长率为空。

空值表示没有可比较的上一期，不应随意填成0。

## 5. NULLIF

NULLIF用于避免分母为0。

如果上月销售额为0，NULLIF会把分母转换为空值，从而避免除以0错误。

## 6. LEAD

LEAD用于取得当前行之后的记录。

结果：

- 1月的下一月销售额为2912.05
- 2月的下一月销售额为6462.00
- 3月没有下一月，因此显示为空

LAG适合回看上一期。

LEAD适合查看下一期。

## 7. 移动平均

本次计算当前月和上一个月的两个月移动平均。

结果：

- 1月：2565.80
- 2月：2738.93
- 3月：4687.02

移动平均可以减弱单月波动，使趋势更平滑。

移动平均不能证明销售额一定持续增长。

## 8. 按渠道计算趋势

先按照月份和渠道进行分组，再使用：

- PARTITION BY channel
- ORDER BY order_month
- LAG(total_net_sales)

这样每个渠道拥有独立的趋势窗口。

京东：

- 1月：758.10
- 2月：696.60，比上一可用月份减少61.50
- 3月：1137.15，比上一可用月份增加440.55

天猫：

- 1月：812.70
- 2月：2215.45，比上一可用月份增加1402.75
- 3月：3802.50，比上一可用月份增加1587.05

抖音：

- 1月：995.00
- 3月：1522.35，比上一可用月份增加527.35

## 9. 上一可用月份与自然上月

LAG取得的是当前查询结果中的上一行。

抖音没有2月记录，因此抖音3月的上一行是1月。

所以抖音3月的变化额不能严格解释为3月环比2月增长额。

如果业务要求严格自然月环比，需要：

1. 生成完整月份列表
2. 为每个渠道补全所有月份
3. 判断缺失月份是零销售还是数据缺失
4. 只有确认零销售后才能填为0
5. 再使用LAG计算相邻自然月变化

## 10. 趋势状态分类

使用CASE WHEN将趋势分为：

- 没有上一可用月份：首次出现
- 本期大于上一可用月份：增长
- 本期小于上一可用月份：下降
- 两期相同：持平

独立练习结果：

- 各渠道首条记录均为首次出现
- 京东2月为下降
- 其他可比较记录为增长

## 11. 分析限制

当前只有3个月数据，样本数量较少。

日期覆盖率只有80%，缺失日期订单可能改变月份分布。

当前没有促销、季节、价格调整等业务背景。

因此，高增长率只能描述当前样本现象，不能证明长期增长趋势或因果关系。

## 12. 今日总结

LAG和LEAD用于取得相邻记录。

移动平均用于平滑短期波动。

PARTITION BY可以让每个渠道独立计算趋势。

使用窗口函数进行时间分析前，必须确认：

- 时间字段是否完整
- 月份是否连续
- 缺失月份表示零销售还是数据缺失
- 指标覆盖率是否足以支持结论
