## 3. CTE基础

CTE可以理解为给一段查询结果临时取一个名字。

基本结构：

    WITH 临时结果名称 AS (
        第一步查询
    )
    SELECT
        字段
    FROM 临时结果名称;

CTE不会永久创建数据库表，只在当前SQL执行期间存在。

CTE适合把复杂分析拆分成多个清晰步骤。

## 4. CTE分析流程

本次使用valid_order_details整理有效订单明细。

第一层负责：

- 关联订单表和商品表
- 筛选有效订单
- 计算单笔模拟成本
- 计算单笔模拟毛利

最终查询负责：

- 使用第一层的结果
- 按商品类别分组
- 计算订单数、销售额、成本和模拟毛利

CTE不会改变业务结果，主要用于改善SQL结构和可读性。

## 5. 多个CTE

一条SQL可以使用多个CTE。

本次分析流程：

原始订单与商品资料
→ valid_order_details有效订单明细
→ channel_summary渠道汇总
→ 最终SELECT渠道分类

多个CTE之间使用逗号分隔。

最后一个CTE结束后不加逗号，直接写最终SELECT。

## 6. CTE字段和别名作用范围

订单表别名s和商品表别名p，只在定义它们的查询层级中有效。

进入下一层CTE后，只能使用上一层CTE输出的字段，不能继续使用上一层内部的表别名。

例如：

第一层输出：

- order_id
- order_month
- net_sales
- estimated_gross_profit

第二层只能使用这些已输出字段。

## 8. 子查询

子查询是在一条SQL内部嵌入另一条SQL。

执行过程通常为：

内部查询先计算结果
→ 外部查询使用内部结果继续筛选或计算

如果内部查询只返回一个值，可以使用等号进行比较。

如果内部查询可能返回多个值，通常需要使用IN。

## 11. CTE与子查询的选择

CTE：

- 适合多步骤分析
- 可以给中间结果命名
- SQL结构更加清晰
- 方便检查每一层的业务含义

子查询：

- 适合临时计算一个值或一小段结果
- 可以直接嵌入WHERE、FROM等位置
- 简单问题通常更加紧凑

CTE和子查询在很多情况下可以互相改写，应根据可读性和分析复杂度选择。

## 12. 常见错误

本次出现的错误包括：

- 最后一个SELECT字段后误加逗号
- 将第一层CTE的表别名带入第二层
- 把SUM聚合函数放入WHERE
- 在简单分组排名中提前使用复杂子查询
- 排序方向与业务要求不一致

处理复杂SQL时，应先让每个步骤能够独立返回正确结果，再组合成完整查询。

## 13. 今日总结

复杂SQL可以按以下方式组织：

数据关联和明细筛选
→ CTE整理中间数据
→ 分组汇总
→ CASE WHEN业务分类
→ HAVING筛选分组
→ 子查询计算比较条件
→ ORDER BY排序
→ 总体指标一致性检查

SQL学习的重点不是背诵整段代码，而是能够根据业务问题确定数据范围、分析步骤和验证方法。
